# Pull Request

現在の変更から PR を作成する。コミットには commit スキルの手順、PR 作成には draft スキルの手順に従う。

## 基本方針

- 変更は**意味的に分離**してコミットする
- 責任範囲が複数にまたがる場合は**マージフローを考慮した PR スタック**として複数の PR に分割する
- 単一の責任範囲に収まる場合は1つの PR で完結させる

## 実行プロセス

### 1. 変更の分析と分割判断

現在の git ステータスと差分を分析し、変更がどの責任範囲に属するかを判断する。

分割の判断基準:

- **パッケージ/サービス境界**: 異なるパッケージやサービスへの変更は別 PR にする
- **レイヤー境界**: リファクタリング → コア実装 → デプロイ設定 → API 層 → フロントエンドのように、レイヤーごとに分離する
- **依存方向**: 下流の変更が上流に依存する場合、上流を先にマージできるようスタックを組む
- **レビュー容易性**: 1つの PR が大きすぎる場合は分割を検討する

分割不要な場合:

- 変更が単一の責任範囲に収まる
- ファイル数が少なく、レビュー負荷が低い
- 依存関係が単純で分割するメリットがない

### 2. PR スタックの設計（複数 PR の場合）

変更を複数 PR に分割する場合、ブランチチェーンを設計する:

```
main ← branch-1 (PR1) ← branch-2 (PR2) ← branch-3 (PR3)
```

- 各 PR は前の PR のブランチを base とする
- マージ順序は PR1 → PR2 → PR3（番号順）
- 先にマージすべき変更（リファクタリング、共通基盤）をスタックの下位に配置する

### 3. コミットの作成

commit スキルの手順に従う。

- PR ごとに対象ファイルをブランチに分けてコミットする
- 各 PR 内でも論理的な単位でコミットを分割する
- Conventional Commits に準拠したメッセージを作成する

### 4. PR の作成

draft スキルの手順に従う。

単一 PR の場合はそのまま作成する。複数 PR の場合は以下を追加で行う:

- 各 PR の body に **PR Stack テーブル**を含める
- スタック内の全 PR のタイトル・番号・状態を一覧化する
- 自身の位置を太字で示す

PR Stack テーブルの形式:

```markdown
## PR Stack

| # | PR | Status |
|---|-----|--------|
| 1 | #123 refactor(xxx): 共通モデルの移動 | Open |
| **2** | **この PR** | |
| 3 | feat(yyy): API 実装 | Next |
```

- `Open`: 既に作成済みの PR
- `この PR`: 現在の PR
- `Next`: この後に作成予定の PR

### 5. スタック内の既存 PR の更新

複数 PR を順次作成する場合、先に作成した PR の body にある PR Stack テーブルを `gh pr edit` で更新し、後続 PR の番号を反映する。
